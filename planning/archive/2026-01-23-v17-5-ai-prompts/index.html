    AI Batch ETL Prompts (Hours + Access Scripts)

    These prompts are for Deep Research + ETL: use the web to verify/fill missing details (especially hours), then return a strict JSON output that can be merged locally.

    Inputs

    Batch files (upload one at a time):

    • docs/audits/v17-5/ai-results/batches/2026-01-21-v17-5-batch1.json (50 services)
    • docs/audits/v17-5/ai-results/batches/2026-01-21-v17-5-batch2.json (50 services)
    • docs/audits/v17-5/ai-results/batches/2026-01-21-v17-5-batch3.json (50 services)
    • docs/audits/v17-5/ai-results/batches/2026-01-21-v17-5-batch4.json (46 services)

    If your batch files don’t include url/address/application_process yet, regenerate them first:

    python scripts/generate-v17-5-ai-batches.py

    Recommended mode

    • ChatGPT Deep Research + Advanced Data Analysis: best combination (web research + downloadable .json output).
    • Gemini Deep Research: works, but you must force JSON-only output (no prose, no markdown).

    Prompt Template (copy/paste)

    TASK TYPE: Deep Research + ETL.

You MAY browse the web and use online research tools. Your goal is to verify or discover the best available public information for each service (especially operating hours), then output strict JSON for automated merging.

You are a Lead Data Engineer. Produce a strictly machine-mergeable enrichment output for the uploaded JSON batch.

Do NOT include narrative in your final output. Do NOT output markdown.

### Input
You have an uploaded JSON file containing an array of service objects. Each service may include:
- id, name, intent_category
- url, phone, email, address
- hours_text
- application_process, documents_required
- eligibility, fees, description
- existing.has_hours, existing.has_access_script

Important:
- Never invent contact details. You may add details only if you found them via web research (and then record the source in `research_sources`).
- Ignore bracket citations like “[48]” if present.
- Preserve IDs exactly. Do not create new IDs or modify existing ones.
- Output items in the same order as the input array.

### Research rules (governance + quality)
- Prefer official sources in this priority order:
  1) the service’s official website (or parent organization’s official site)
  2) municipal/provincial/federal government pages
  3) reputable local directories (e.g., United Way / 211) when official pages are unclear
- Do not scrape behind logins or paywalls.
- If sources conflict, prefer the most recent official source and record the conflict in `research_sources`.
- Do not copy large quotes; use short paraphrases when necessary.

### Transformations

#### 1) Hours Parsing (output field: `hours`)
Goal: Convert `hours_text` into a structured weekly schedule object compatible with our schema:
- `hours` is either `null` (unknown/unparseable) OR an object with:
  - optional day keys: monday..sunday, each as `{ "open": "HH:MM", "close": "HH:MM" }`
  - optional `notes` (string)

Rules:
- Time format must be 24-hour **HH:MM** with leading zeros (e.g., "09:00", "17:30").
- If a day is closed/unknown, OMIT that day key (do not use null open/close).
- For “24/7” services: include all 7 days as `{ "open": "00:00", "close": "23:59" }`.
- For split schedules (e.g., "9-12, 1-5"): use earliest open and latest close; put the mid-day closure in `notes`.
- If `hours_text` is empty/null or clearly incomplete: use web research to find official hours. If you can’t confidently determine hours, `hours: null`.
- If the text says “By appointment”: set `hours` to `{ "notes": "By appointment only" }`.
- If multiple channels are listed (phone/text/chat): prefer the broadest “main availability” and add details to `notes`.
- If timezones are present (e.g., “ET”): keep the hours as written and note the timezone in `notes`.
- If parsing is ambiguous, choose `hours: null` rather than guessing.

#### 2) Access Script Generation (output field: `access_script`)
Goal: Generate a short, administrative “how to access” script.

Constraints:
- 1–3 sentences, Grade 6–8 readability, direct and objective.
- Include ONLY access mechanics (call / visit / email / website), required docs, fees, and eligibility.
- Never include emotional support language or service marketing.
- Never invent addresses, phone numbers, emails, fees, eligibility, hours, or referral requirements.

Channel priority:
1) phone
2) url
3) email
4) address (only if present)

Crisis rule:
- If `intent_category` is "Crisis" and you produce an access script, append: "If you are in immediate danger, call 911."

Skip rule:
- If `existing.has_hours` is true, output `"hours": null` (do not replace existing structured hours).
- If `existing.has_access_script` is true, output `"access_script": null` (do not replace existing curated scripts).
- If there is no usable contact method (no phone/url/email/address) AND you cannot find a reliable one via research, output `"access_script": null`.

### Output
Return valid JSON in ONE of these two formats:

Option A (preferred): a JSON object with:
- `processed_batch_output`: the enrichment array (used by our merge script)
- `research_sources`: source tracking for governance and spot-checking

Option B: the enrichment array only.

The enrichment array items MUST be:
{
  "id": "<original id>",
  "hours": <object or null>,
  "access_script": <string or null>
}

### Hard requirements
- Output must be valid JSON ONLY. No markdown code fences. No commentary.
- Every input `id` must appear exactly once in the output.
- If you cannot parse something reliably, use `null` instead of guessing.

### Validation (must run before you respond)
Validate these rules across all output rows:
- Output is either:
  - a JSON array (the enrichment array), OR
  - a JSON object with `processed_batch_output` (enrichment array) and `research_sources` (array).
- Every output item has exactly 3 keys: id, hours, access_script.
- No duplicate IDs; output count equals input count.
- Any time values match `^([0-1]?[0-9]|2[0-3]):[0-5][0-9]$` and are normalized to leading-zero HH:MM (e.g., "9:00" -> "09:00").
- Never emit `{ "open": null, "close": null }` day objects. Omit the day key instead.

### research_sources format (if using Option A)
`research_sources` MUST be an array of objects:
{
  "id": "<original id>",
  "sources": [
    {
      "url": "https://...",
      "what_verified": "Hours / access method / phone number / etc.",
      "confidence": "high" | "medium" | "low"
    }
  ],
  "notes": "Optional: conflicts, ambiguity, timezone, seasonal/holiday caveats"
}

If you can execute code, write a small script to:
- load the uploaded JSON file,
- build the output list,
- validate JSON + time formats,
- and save it as `processed_batch_output.json`.

If you cannot execute code, still return ONLY valid JSON (either Option A or Option B). Do not add any other text before or after the JSON.

    After Getting Outputs

    Save each model output as:

    • batch1_output.json
    • batch2_output.json
    • batch3_output.json
    • batch4_output.json

    Then merge:

    npx tsx scripts/merge-ai-enrichment.ts batch1_output.json batch2_output.json batch3_output.json batch4_output.json

    Quick sanity checks before merging:

    python -m json.tool batch1_output.json > /dev/null
python -m json.tool batch2_output.json > /dev/null
python -m json.tool batch3_output.json > /dev/null
python -m json.tool batch4_output.json > /dev/null

    If you explicitly want to overwrite existing data:

    npx tsx scripts/merge-ai-enrichment.ts --overwrite-hours --overwrite-access-script batch1_output.json batch2_output.json batch3_output.json batch4_output.json

    Translation Prompt (Access Script → French)

    Use this when you want to populate access_script_fr from the existing English access_script.

    Important: This is translation-only (no web research, no new facts).

    Inputs

    • Generate batches with:
    npm run export:access-script-fr
    • Upload one file at a time from:
    • docs/audits/v17-5/ai-results/access-script-fr/input/batch-*.input.json

    Prompt Template (copy/paste)

    TASK TYPE: Translation-only ETL (NO web research).

You are translating short administrative scripts from English to French.
You MUST NOT browse the web. You MUST NOT add new factual claims (hours, eligibility, fees, documents, referral rules, addresses, phone numbers, emails).

### Input
You have an uploaded JSON file with:
{
  "batch": <number>,
  "total_batches": <number>,
  "items": [
    { "id": "...", "name": "...", "name_fr": "...", "scope": "...", "access_script": "..." }
  ]
}

Rules:
- Preserve `id` exactly (no edits).
- Translate `access_script` into idiomatic Canadian French.
- Keep phone numbers, URLs, and organization names unchanged (translate only the surrounding words).
- Keep the tone simple and administrative (Grade 6–8).
- Do not add “call 911” unless it already exists in the English script; if present, translate it.
- Output items in the SAME order as input.

### Output (strict JSON only)
Return Option A:
{
  "processed_batch_output": [
    { "id": "<id>", "access_script_fr": "<translated french script>" }
  ],
  "research_sources": []
}

Hard requirements:
- Valid JSON only. No markdown. No commentary.
- Every input `id` appears exactly once.
- `access_script_fr` must be a non-empty string for every item.

Validation (must run before you respond):
- Output is a JSON object with `processed_batch_output` (array) and `research_sources` (empty array).
- Count of `processed_batch_output` equals count of input `items`.
- No duplicate IDs.
